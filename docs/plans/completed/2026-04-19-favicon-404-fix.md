# Fix `/favicon.ico` 404 on docketworks instances

## Context

Browsers requesting `https://office.morrissheetmetal.co.nz/favicon.ico` (and every other docketworks instance) get a 404. The favicon file itself is fine — `frontend/public/favicon.ico` (15,406 bytes) is copied by Vite into `frontend/dist/favicon.ico` on `npm run build`. The bug is in the nginx template.

### Root cause

`scripts/server/templates/nginx-instance.conf.template:5`:

```nginx
location = /favicon.ico { access_log off; log_not_found off; }
```

This exact-match location matches `/favicon.ico` with higher priority than the catch-all `location /` block at line 33 (which has `root /opt/docketworks/instances/__INSTANCE__/frontend/dist` and would otherwise serve the file). The exact-match block has no `root`, `alias`, or file handler — so nginx returns 404.

Two wrongs in one line: the file isn't served, *and* `access_log off; log_not_found off;` silence any record that the request ever arrived or that a file was missing. That violates the project's fail-early / no-fallbacks philosophy — noise is signal, and silencing it destroys the signal.

## Fix

**Delete line 5 entirely.** No replacement.

With the exact-match block gone, the `location /` block at line 33-36 takes effect for `/favicon.ico`:

```nginx
location / {
    root /opt/docketworks/instances/__INSTANCE__/frontend/dist;
    try_files $uri $uri/ /index.html;
}
```

`$uri` = `/favicon.ico`, so nginx resolves `/opt/docketworks/instances/__INSTANCE__/frontend/dist/favicon.ico`, the file exists, it's served — logged normally. If the file ever disappears, the 404 lands in the access log where we can see it.

## Files to modify

- **`scripts/server/templates/nginx-instance.conf.template`** — delete line 5 and its surrounding blank line so the `location /api/` block remains the first location directive.

## Deployment — already-live instances

`scripts/server/instance.sh:422-428` only applies the template during instance **creation** (no re-apply command). Every already-generated `/etc/nginx/sites-available/docketworks-<instance>` on the server was written from the old template and still has the broken block.

For each live instance on the `/opt/docketworks/` layout (e.g. `msm-prod` once migrated per `project_prod_migration.md`; all UAT subdomains per `project_uat_simplified.md`):

1. SSH to the server; stay on `main` (per `feedback_server_main_only.md`).
2. `sudo vi /etc/nginx/sites-available/docketworks-<instance>` and delete the `location = /favicon.ico { ... }` line.
3. `sudo nginx -t && sudo systemctl reload nginx`.
4. Verify: `curl -I https://<fqdn>/favicon.ico` → `200 OK`, `Content-Length: 15406`.

Any instance still on the legacy `/home/django_user/jobs_manager_front/dist` layout is out of scope — the prod migration plan handles that path.

## Verification

- `grep -n favicon scripts/server/templates/nginx-instance.conf.template` → no matches.
- Diff shows only line 5 removed.

### On the server after rollout
- `curl -I https://office.morrissheetmetal.co.nz/favicon.ico` → `200 OK`, `Content-Length: 15406`.
- `curl -I https://<any-uat-subdomain>.docketworks.site/favicon.ico` → `200 OK`.
- Browser DevTools on the page: no 404 for `favicon.ico` in Console/Network.
- `sudo tail -f /var/log/nginx/docketworks_<instance>_access.log` during a page load — favicon request is now recorded (as it should be).

## Non-goals

- Not touching `restore/extracted/` nginx configs — legacy backup archive, not live.
- Not adding Django STATIC_URL/STATIC_ROOT — Django is API-only; nginx serves all static assets from `frontend/dist`.
- Not changing the favicon image itself.
