# Production MySQL → PostgreSQL Migration Runbook

CRITICAL: audit

Every command and its key output must be logged, same as the backup-restore process.
e.g. `logs/mysql_to_postgres_migration_YYYYMMDD.log`

## Overview

One-time migration of production from MariaDB to PostgreSQL. This runbook was tested via a dev dry run using a production backup. Follow it step by step — do not skip steps or run out of order.

## Principles

- Preserve exact UUIDs/PKs — no `--natural-foreign`, no `--natural-primary`
- Plain `dumpdata` / `loaddata` — nothing clever
- Keep MariaDB untouched throughout — rollback is "switch .env back"
- Verify row counts at every checkpoint

## What gets excluded from the dump (and why it's safe)

| Excluded | Why | How regenerated |
|---|---|---|
| `contenttypes` | Auto-generated by Django from installed models. Integer PKs differ between databases but carry no meaning. | `manage.py migrate` creates them |
| `auth.permission` | Auto-generated from model Meta permissions. Same as contenttypes. | `manage.py migrate` creates them |

No `GenericForeignKey` usage in this codebase, so content type PK differences don't affect any FK references.

## Pre-Migration Checklist

- [ ] This runbook has been tested via dev dry run with production backup
- [ ] PostgreSQL installed and running on the server
- [ ] `scripts/setup_database.sh` tested on the server
- [ ] Downtime window communicated (estimate: 30-60 minutes)
- [ ] `mysqlclient` Python package still in venv (needed for the dump step)
- [ ] Sufficient disk space for the JSON dump file (estimate: 2-3x database size)
- [ ] `.env` has `DB_ENGINE=django.db.backends.mysql` (or no `DB_ENGINE` — must be pointing at MariaDB for Step 2)

## Migration Steps

Commands below use UAT conventions. Adjust paths for your environment:
- `$INSTANCE` — e.g. `msm`
- `$ENV_FILE` — e.g. `/opt/docketworks/instances/msm/.env`
- `$CODE_DIR` — e.g. `/opt/docketworks/instances/msm/code`

Read DB credentials from .env before starting:
```bash
export DB_NAME=$(grep -E '^DB_NAME=' "$ENV_FILE" | cut -d= -f2)
export DB_USER=$(grep -E '^DB_USER=' "$ENV_FILE" | cut -d= -f2)
export DB_PASSWORD=$(grep -E '^DB_PASSWORD=' "$ENV_FILE" | cut -d= -f2)
```

### Step 1: Stop the application

```bash
sudo systemctl stop gunicorn-$INSTANCE
```

**Check:** `systemctl status gunicorn-$INSTANCE` shows inactive.

### Step 2: Capture MariaDB row counts (baseline)

```bash
python manage.py shell -c "
from django.apps import apps
total = 0
for model in sorted(apps.get_models(), key=lambda m: m._meta.label):
    count = model.objects.count()
    if count > 0:
        print(f'{model._meta.label}: {count}')
        total += count
print(f'TOTAL: {total}')
" | tee /tmp/dw_mysql_counts.txt
```

**Check:** Numbers look reasonable. Save this output — it's the baseline for Step 9.

### Step 3: Dump data from MariaDB

The .env must still point at MariaDB for this step.

```bash
python manage.py dumpdata \
    --exclude contenttypes \
    --exclude auth.permission \
    --indent 2 \
    --output /tmp/dw_mysql_to_pg.json
```

**Check:**
```bash
ls -lh /tmp/dw_mysql_to_pg.json
# Expect 50-200MB
```

### Step 4: Create PostgreSQL database

```bash
./scripts/setup_database.sh --db "$DB_NAME" --user "$DB_USER" --password "$DB_PASSWORD" --drop
```

**Check:** Output shows "Database $DB_NAME ready."

### Step 5: Switch .env to PostgreSQL

Edit the instance .env file:

```
DB_ENGINE=django.db.backends.postgresql
```

Or remove `DB_ENGINE` entirely (default is PostgreSQL).

### Step 6: Run migrations on PostgreSQL

```bash
python manage.py migrate --no-input
```

**Check:** All migrations show "OK". No errors.

### Step 7: Truncate auto-generated content types

`migrate` creates content type records. We need to clear them before loading the dump, because the dump contains all business data that was serialized with MySQL's content type PKs excluded.

```bash
PGPASSWORD="$DB_PASSWORD" psql -h 127.0.0.1 -U "$DB_USER" "$DB_NAME" \
    -c "TRUNCATE django_content_type CASCADE;"
```

**Check:** Output shows "TRUNCATE TABLE".

### Step 8: Load data into PostgreSQL

```bash
python manage.py loaddata --no-transactions /tmp/dw_mysql_to_pg.json
```

**Check:** Shows "Installed NNNNN object(s) from 1 fixture(s)". No errors.

### Step 9: Verify row counts

```bash
python manage.py shell -c "
from django.apps import apps
total = 0
for model in sorted(apps.get_models(), key=lambda m: m._meta.label):
    count = model.objects.count()
    if count > 0:
        print(f'{model._meta.label}: {count}')
        total += count
print(f'TOTAL: {total}')
" | tee /tmp/dw_pg_counts.txt
```

**Compare:**
```bash
diff /tmp/dw_mysql_counts.txt /tmp/dw_pg_counts.txt
```

**Expected:** All business model counts identical. The only differences should be:
- `auth.Permission` — regenerated by migrate (count should still match)
- `contenttypes.ContentType` — regenerated by migrate (count should still match)

If any business model count differs: **STOP. Do not proceed. Investigate.**

### Step 10: UUID spot-check

Pick a few known records and verify UUIDs match between MySQL and PostgreSQL:

```bash
PGPASSWORD="$DB_PASSWORD" psql -h 127.0.0.1 -U "$DB_USER" "$DB_NAME" -c "
SELECT id, job_number, name FROM job_job ORDER BY job_number DESC LIMIT 5;
"
```

Compare against values from the MariaDB database (still running, still has the data).

### Step 11: Django health checks

```bash
python manage.py check
python manage.py showmigrations | grep '\[ \]'
```

**Expected:** `check` passes. `showmigrations` returns no output (all migrations applied).

### Step 12: Start the application

```bash
sudo systemctl start gunicorn-$INSTANCE
```

**Check:** `systemctl status gunicorn-$INSTANCE` shows active.

### Step 13: Smoke test

- [ ] Log in to the application
- [ ] View the Kanban board — jobs display correctly
- [ ] Open a job detail page — cost lines, events load
- [ ] Check Xero sync status — no errors
- [ ] Create a test timesheet entry
- [ ] Verify a client page loads with contacts

## Rollback Procedure

MariaDB is untouched throughout this process. To roll back:

1. `sudo systemctl stop gunicorn-$INSTANCE`
2. Edit .env: set `DB_ENGINE=django.db.backends.mysql` (or remove `DB_ENGINE`)
3. `sudo systemctl start gunicorn-$INSTANCE`

Application is back to pre-migration state. No data lost.

## Post-Migration Cleanup

Once PostgreSQL is verified and running stably (recommend: 1 week minimum):

- [ ] Remove `mysqlclient` from Python dependencies
- [ ] Remove MariaDB: `sudo apt remove --purge mariadb-server`
- [ ] Remove MySQL-specific migration vendor checks (migrations 0024, 0030)
- [ ] Delete dump file: `rm /tmp/dw_mysql_to_pg.json`
- [ ] Update `backport_data_backup` to remove `contenttypes` from `INCLUDE_MODELS` and remove manual `migrations.migration` serialization
- [ ] Remove `DB_ENGINE` env var support from settings.py (hardcode PostgreSQL)
